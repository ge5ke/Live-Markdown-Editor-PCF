# v3 Overhaul — Implementation Plan

Executes every decision in `docs/INVESTIGATION.md` (decision log entries 1–10, items A–N).
Authoritative source for *why*: that document. This plan is the authoritative source for *what*.

## Global Constraints

- TypeScript must compile: `npm run build` passes with zero errors after every task.
- Lint must pass: `npm run lint` clean after every task.
- There is NO test infrastructure and adding one is out of scope (deferred by decision).
  Verification per task = successful build + lint + the specific manual checks named in the task.
- No new runtime dependencies beyond those explicitly named in a task.
- No `console.log` left behind. Errors go through `handleError` from `utils/errorHandler`.
- Never reintroduce anything a prior task removed (Find & Replace, save status, theme
  machinery, `window.prompt`-based flows once replaced).
- PCF lifecycle rules: `getOutputs()` must always return current `_currentValue`;
  `destroy()` must never lose data; `updateView` may be called at any time with any values.
- The control must work with multiple instances on one page: no `document.getElementById`,
  no global mutable state keyed by fixed ids.
- Match existing code style (4-space indent, existing naming conventions).
- Commit at the end of each task with a descriptive message.

## Task 2: Value-flow rework — notify-on-blur, flush on destroy, external updates

**Goal:** Implement decision 1 (notify-on-blur), decision 2/3 fixes (first-updateView
clobber, external update policy), and kill the per-keystroke re-render echo.

**New value-flow model:**

Inside `MarkdownEditor.tsx` (`EditorComponent`):
- Keep the 150 ms serialize debounce for updating `currentMarkdownRef` and the stats
  display, but **do NOT call `onUpdate` from the serialize listener anymore**.
- Add `isDirtyRef` (boolean): set `true` in the Milkdown `updated` listener (synchronous
  part), cleared on flush.
- Add a `flush()` function (useCallback, stable via refs): if a serialize timeout is
  pending, clear it and serialize the current doc synchronously (get `serializerCtx` +
  current doc via `editorViewCtx`); then if `isDirtyRef.current`, call
  `onUpdate(currentMarkdownRef.current)` and set `isDirtyRef.current = false`.
- Attach a `focusout` listener on the container div. On focusout, if
  `event.relatedTarget` is null or NOT contained in the container element, call `flush()`.
  (Moving focus from the editor to a toolbar button must NOT flush.)
- Unmount cleanup (`useEffect` return) must call `flush()` BEFORE clearing timers, so
  React unmount → flush → `onUpdate` → shell notify happens synchronously inside
  `root.unmount()`.
- **External value application:** add an effect on `[value]` (rename the prop back from
  `initialValue` to `value` for honesty): when `value !== currentMarkdownRef.current`
  AND `isDirtyRef.current === false`, parse `value` (empty string included — replace with
  an empty doc) and replace the whole document, then set
  `currentMarkdownRef.current = value` and refresh stats. This REPLACES the deleted
  "empty editor resync" effect and must handle `value === ""` (use the parser on `""`;
  if the parser returns a falsy doc for empty input, build an empty doc from
  `state.schema.topNodeType.createAndFill()`).
- The Milkdown editor is still created once with the first `value` as `defaultValueCtx`.

In `index.ts` (`MarkdownEditorControl`):
- `handleChange(value)`: set `_currentValue = value`, recompute word/char counts, set
  `_isValid`, and call `this._notifyOutputChanged()` **synchronously** (delete the 50 ms
  debounce, `_notifyTimeoutId`, and its cleanup in `destroy`). `handleChange` is now only
  invoked on flush (blur/unmount), so eager notification is correct.
- `updateView`: delete `_hasUserEdited`, `_initialLoadComplete`, and `_lastPropsSignature`
  (the `JSON.stringify` signature). New logic:
  - Read `newValue = context.parameters.value?.raw ?? ""` (nullish — empty string is a
    legitimate value, decision 3).
  - Track previous rendered props in individual private fields (`_lastReadOnly`,
    `_lastShowToolbar`, etc. — plain comparisons, no JSON).
  - If `newValue !== this._currentValue`, set `this._currentValue = newValue` and
    re-render (the component's dirty-guard decides whether to apply it — the shell does
    not need to know about dirtiness).
  - Re-render when any other prop changed. Skip rendering when nothing changed.
- `destroy()`: call `this._root.unmount()` (which flushes via the component) — the
  notify path must still work during destroy, so unmount BEFORE nulling
  `_notifyOutputChanged`-related state. No timer clearing needed anymore.

**Edge cases that must hold (reviewer: check these in the code):**
1. User types, clicks a toolbar button → no flush (focus stayed inside container).
2. User types, clicks elsewhere on the form → flush → notify → updateView echo arrives
   with the same value → `newValue === _currentValue` → no re-render.
3. User clears all content, blurs → `onUpdate("")` → output is `""`.
4. Form refresh with user mid-edit (dirty) → new value stored in `_currentValue` and
   passed down, but component skips applying it (dirty guard). NOTE consequence: shell
   `_currentValue` now differs from editor content until next flush overwrites it — this
   is accepted (platform sees its own value back only until the user blurs).
5. Control destroyed with pending edits → unmount flush delivers them.

**Verification:** build + lint pass. Trace each edge case in the final code and state in
the report where each is satisfied.

## Task 5: maxLength hard-block, honest counts, per-instance stats

**Goal:** Decision 4 (hard-block) + investigation items 13 (count mismatch) and J
(word-count accuracy) + multi-instance id fix.

- **Metric definition (single source of truth):** character count = `doc.textContent.length`
  (what the user sees). Word count = matches of `/\S+/g` against `doc.textContent`
  (not the serialized markdown). `index.ts` output `characterCount`/`wordCount` must use
  the SAME numbers: compute them in the component at flush time and pass them through —
  change the `onUpdate` callback signature to `(markdown: string, stats: { words: number; chars: number })`
  and have `index.ts.handleChange` store them instead of recomputing from markdown.
- **Hard block:** add a ProseMirror `filterTransaction` (via a small `$prose` plugin —
  import `$prose` from `@milkdown/utils`, `Plugin`/`PluginKey` from `@milkdown/prose/state`):
  reject any transaction where `tr.docChanged && tr.doc.textContent.length > maxLength`.
  Keep `maxLength` readable at filter time via a ref (prop can change). KNOWN LIMIT (add
  as code comment): the enforced metric is text length; serialized markdown adds syntax
  overhead, so makers must set maxLength below the Dataverse column limit with headroom.
- `isValid` output: with the hard block active this stays `true`; keep computing it
  (`chars <= maxLength`) for backward compatibility of the output property.
- **Per-instance stats DOM:** replace `document.getElementById('md-word-count'/'md-char-count')`
  with React refs (`wordCountElRef`, `charCountElRef`) attached to the status-bar spans;
  update `ref.current.textContent` directly. Remove the `id` attributes.
- Remove the now-duplicated stats regex/logic from `index.ts` (`WORD_MATCH_REGEX` usage
  there) — stats arrive via the callback.

**Verification:** build + lint. Report must state where the filterTransaction lives and
how `maxLength` prop changes reach it.

## Task 7: URL validation hardening

**Goal:** Investigation item 17 — close the whitespace/control-char protocol bypass.

- Rewrite `validateLinkUrl` / `validateImageUrl` in `utils/security.ts` around real URL
  parsing: strip ASCII control chars and whitespace (`/[ - ]/g` INSIDE the
  string, matching browser URL normalization) before analysis; then `new URL(candidate)`
  — if it throws, try `new URL(candidate, 'https://placeholder.invalid')` to classify it
  as relative. Validate `parsed.protocol` against allowlists: links
  `http: https: mailto: tel: ftp:`; images `http: https:` plus `data:` ONLY when the
  full href starts with `data:image/` after stripping. Relative URLs (base fallback used)
  stay valid, returned as originally trimmed. `DANGEROUS_PROTOCOLS` blocklist constant
  becomes redundant — delete it (allowlist is the mechanism); keep
  `SAFE_LINK_PROTOCOLS`/`SAFE_IMAGE_PROTOCOLS` as the allowlists.
- The `sanitized` value returned must be the cleaned string that was validated (control
  chars stripped), and callers must insert `sanitized`, never the raw input — verify both
  call sites (link popover, image popover/paste from Task 6).
- Include in the code a short comment listing the attack shapes covered:
  `java\tscript:`, `javascript:`, mixed-case, protocol-relative.

**Verification:** build + lint. In the report, table of ~8 sample inputs → expected
verdict (including `java\tscript:alert(1)` → invalid, `https://x` → valid, `/rel` →
valid, `data:image/png;base64,x` → valid for images, `data:text/html,x` → invalid).
